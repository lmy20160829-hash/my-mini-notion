## User Scenarios & Testing *(mandatory)*

### User Story 1 - 글 상세에서 커버 이미지를 본다 (Priority: P1)

사용자가 글 상세 화면을 열면 제목 입력창 바로 위에 커버 이미지 영역이 나타나고, 랜덤 고양이 사진이 표시된다. 커버 이미지는 글 상단을 시각적으로 채워 노션 문서처럼 문서에 개성과 분위기를 더한다.

**Why this priority**: 이 기능의 핵심 가치 자체가 "제목 위 커버 이미지"이다. 이미지가 표시되는 것만으로 최소 기능(MVP)이 성립하며, 나머지 스토리는 이 경험을 다듬는 역할이다.

**Independent Test**: 아무 글이나 상세 화면으로 열었을 때 제목 입력창 위쪽에 커버 이미지가 렌더링되고 실제 고양이 사진이 보이면 통과. 다른 스토리(스켈레톤·실패 처리) 없이도 단독으로 가치를 전달한다.

**Acceptance Scenarios**:

1. **Given** 사용자가 목록에서 글 하나를 선택했을 때, **When** 글 상세 화면이 열리면, **Then** 제목 입력창 위에 커버 이미지 영역이 나타나고 랜덤 고양이 사진이 표시된다.
2. **Given** 커버 이미지가 표시된 상태에서, **When** 사용자가 화면을 확인하면, **Then** 커버 이미지는 제목 입력창보다 위(상단)에 위치한다.
3. **Given** 사용자가 다른 글로 이동했을 때, **When** 새 글 상세가 열리면, **Then** 그 글에도 커버 이미지 영역이 동일하게 나타난다.

---

### User Story 2 - 로딩 중에는 스켈레톤으로 자리를 채운다 (Priority: P2)

커버 이미지가 아직 도착하지 않은 동안, 사용자는 회전하는 로딩 스피너가 아니라 커버 영역 모양의 스켈레톤(자리표시자)을 본다. 이미지가 준비되면 스켈레톤이 실제 사진으로 대체된다.

**Why this priority**: 이미지 표시(P1)가 동작한 뒤, 로딩 경험을 매끄럽게 만드는 개선이다. 사용자가 요청한 명시적 요구사항이지만 이미지 표시 자체가 선행되어야 의미가 있다.

**Independent Test**: 네트워크를 느리게 한 상태(또는 로딩 상태를 강제)에서 글 상세를 열면, 커버 영역에 스피너가 아닌 스켈레톤 형태의 자리표시자가 보이고, 이미지 도착 후 사진으로 바뀌면 통과.

**Acceptance Scenarios**:

1. **Given** 커버 이미지 요청이 아직 완료되지 않은 상태에서, **When** 글 상세 화면이 표시되면, **Then** 커버 영역에 스켈레톤 자리표시자가 보이고 로딩 스피너는 보이지 않는다.
2. **Given** 스켈레톤이 표시된 상태에서, **When** 커버 이미지 로딩이 완료되면, **Then** 스켈레톤이 실제 고양이 사진으로 대체된다.
3. **Given** 스켈레톤이 표시되는 동안, **When** 커버 영역이 렌더링되면, **Then** 스켈레톤은 최종 커버 이미지와 동일한 크기·위치를 차지해 이미지 도착 시 레이아웃이 크게 튀지 않는다.

---

### User Story 3 - 이미지를 못 불러와도 화면이 깨지지 않는다 (Priority: P3)

고양이 사진 API가 응답하지 않거나 이미지를 불러오지 못하는 경우에도, 사용자는 깨진 이미지 아이콘이나 무너진 레이아웃 대신 커버 공간을 보존하는 중립 자리표시자를 본다. 커버 이미지가 없더라도 제목·본문 편집은 정상 동작한다.

**Why this priority**: 안정성 보강이다. 외부 오픈 API에 의존하므로 실패는 반드시 발생할 수 있지만, 핵심 편집 흐름을 막지 않으므로 우선순위는 낮다.

**Independent Test**: API 요청이 실패하도록 유도(오프라인/차단)한 뒤 글 상세를 열었을 때, 깨진 이미지 아이콘 없이 정돈된 커버 영역이 유지되고 제목·본문 입력이 정상 동작하면 통과.

**Acceptance Scenarios**:

1. **Given** 고양이 사진 API가 응답하지 않는 상태에서, **When** 글 상세 화면이 열리면, **Then** 네이티브 오류 시 즉시, 오류 없이 지연되면 약 10초 이내에 스켈레톤이 종료되고, 깨진 이미지 아이콘이나 무너진 레이아웃 없이 커버 공간을 보존하는 중립 자리표시자가 표시된다(제목·본문 위치는 이동하지 않는다).
2. **Given** 커버 이미지를 불러오지 못한 상태에서, **When** 사용자가 제목이나 본문을 편집하면, **Then** 편집 기능은 커버 실패와 무관하게 정상 동작한다.

---

### Edge Cases

- **느린 네트워크**: 이미지 도착이 지연되면 스켈레톤이 계속 유지되고, 도착 시 사진으로 대체된다.
- **로딩 완료 전 화면 이탈**: 이미지가 도착하기 전에 사용자가 목록으로 돌아가거나 다른 글로 이동하면, 진행 중이던 로딩 상태가 화면을 방해하지 않는다.
- **API 실패/타임아웃**: 네이티브 이미지 오류 시 즉시 실패로 전환하고, 오류 없이 지연되는 경우 약 10초 상한 타임아웃을 초과하면 스켈레톤을 종료하고 정돈된 실패 상태로 마무리한다 — 무한 대기하지 않는다(User Story 3).
- **같은 글 재방문**: 글을 다시 열면 커버 영역이 다시 나타난다(랜덤 특성상 이전과 다른 고양이 사진일 수 있다 — Assumptions 참조).

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 시스템은 글 상세 화면에서 제목 입력창 **위**에 커버 이미지 영역을 표시해야 한다(MUST).
- **FR-002**: 커버 이미지는 외부 랜덤 고양이 사진 오픈 API(엔드포인트: `https://cataas.com/cat/cute`)에서 가져와야 한다(MUST).
- **FR-003**: 커버 이미지 로딩 중에는 로딩 스피너가 아니라 스켈레톤 UI(자리표시자)를 표시해야 한다(MUST).
- **FR-004**: 커버 이미지 로딩이 완료되면 스켈레톤을 실제 이미지로 대체해야 한다(MUST).
- **FR-005**: 스켈레톤은 최종 커버 이미지와 동일한 영역(크기·위치)을 점유해, 이미지 도착 시 레이아웃 이동(layout shift)을 최소화해야 한다(MUST).
- **FR-006**: 커버 이미지를 불러오지 못한 경우, 시스템은 깨진 이미지 아이콘이나 무너진 레이아웃을 노출하지 않고, 커버 공간을 그대로 보존하는 중립 자리표시자(`DESIGN.md` 톤의 은은한 배경/아이콘)를 유지해야 한다(MUST). 커버 영역을 숨겨 제목·본문이 위로 이동하게 해서는 안 된다.
- **FR-007**: 커버 이미지의 표시·로딩·실패 상태와 무관하게 제목·본문 편집 기능은 정상 동작해야 한다(MUST).
- **FR-008**: 커버 이미지 영역은 프로젝트의 기존 디자인 시스템(`DESIGN.md`의 토큰·간격·라운드·모션)과 시각적으로 일관되어야 한다(MUST).
- **FR-009**: 시스템은 커버 이미지 로딩이 네이티브 오류를 반환하면 즉시 실패 상태로 전환하고, 오류 없이 약 10초 상한 타임아웃을 초과하면 스켈레톤을 종료하고 실패 상태로 전환해야 한다(MUST) — 스켈레톤에서 무한 대기하지 않는다.

### Key Entities *(include if feature involves data)*

- **커버 이미지(Cover Image)**: 글 상세 화면 상단에 표시되는 **일시적(transient) 표시용 리소스**. 외부 오픈 API에서 매 요청 시 가져오며, 별도로 영구 저장하지 않는다. 상태는 "로딩 중(스켈레톤)", "표시됨(이미지)", "실패(커버 공간을 보존하는 중립 자리표시자)" 세 가지를 가지며, 세 상태 모두 동일한 커버 영역(크기·위치)을 점유한다.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 글 상세 화면을 여는 모든 경우에 제목 입력창 위 커버 영역이 나타나며, 사용자는 다음 세 상태 중 하나만 본다 — 스켈레톤, 실제 사진, 또는 정돈된 실패 상태(깨진 이미지 아이콘은 0건).
- **SC-002**: 커버 이미지가 아직 도착하지 않은 동안 사용자는 즉시(체감상 지연 없이) 스켈레톤 자리표시자를 보며, 로딩 스피너는 어떤 경우에도 나타나지 않는다.
- **SC-003**: 스켈레톤이 실제 이미지로 대체될 때 커버 아래의 제목·본문 위치가 눈에 띄게 튀지 않는다(레이아웃 이동 최소화).
- **SC-004**: 커버 이미지 로딩이 실패하거나 지연되어도 사용자는 제목·본문 편집을 100% 완료할 수 있다(편집 흐름이 차단되지 않는다).
- **SC-005**: 커버 이미지가 도착하지 않는 경우에도 사용자는 최대 약 10초 이내에 스켈레톤이 아닌 확정 상태(정돈된 실패 처리)를 보게 되며, 무한 로딩 상태에 머무르지 않는다.

## Assumptions

- **랜덤 갱신 정책**: 커버 이미지는 글 상세 화면을 열 때마다 API에서 새로 가져온다. 별도 영구 저장이나 글별 고정을 하지 않으므로, 같은 글을 다시 열면 다른 고양이 사진이 표시될 수 있다(가장 단순한 해석, YAGNI 원칙 준수). 글별로 커버를 고정 저장하는 것은 이번 범위에서 제외한다.
- **표시 전용(비편집)**: 커버 이미지는 사용자가 직접 업로드/교체/제거하지 않는 표시 전용 요소다. 커버 변경 UI, 사용자 업로드, 커버 위치 조정은 이번 범위에서 제외한다.
- **적용 범위**: 커버 이미지는 글 상세 화면(`app/(app)/posts/[id]`)에만 적용한다. 목록·마이페이지·로그인 등 다른 화면은 범위 밖이다.
- **외부 의존성**: 이 기능은 외부 오픈 API(`https://cataas.com/cat/cute`)의 가용성에 의존한다. API가 불안정하면 실패 상태(User Story 3)로 처리한다.
- **인증/권한**: 기존 앱의 접근 흐름을 그대로 사용하며, 커버 이미지는 추가 인증을 요구하지 않는 공개 오픈 API를 사용한다.

## Dependencies

- 외부 서비스: cataas.com 랜덤 고양이 사진 API (`https://cataas.com/cat/cute`).
- 기존 글 상세 화면(제목 입력창을 포함한 `app/(app)/posts/[id]/page.tsx`).
- 프로젝트 디자인 시스템 문서 `DESIGN.md`(커버 영역·스켈레톤의 토큰·간격·모션 정의는 구현 단계에서 여기에 반영).
