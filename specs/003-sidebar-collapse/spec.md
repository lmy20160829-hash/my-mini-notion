# Feature Specification: 사이드바 접기/펼치기

**Feature Branch**: `003-sidebar-collapse`

**Created**: 2026-07-16

**Status**: Draft

**Input**: User description: "사이드바를 추가하고, 접기/펼치기 기능을 구현해줘. 사이드바에 버튼 하나 만들고, 이걸 클릭하면 접기, 다시 클릭하면, 펼치기가 되도록 해줘."

## Clarifications

### Session 2026-07-16

- Q: 사이드바를 접었을 때 어떤 모습이어야 하나? (다시 펼치는 버튼 위치 포함) → A: **좁은 아이콘 레일로 축소.** 접어도 아이콘만 남는 좁은 레일이 유지되고, 같은 토글 버튼 1개가 상시 표시되어 양방향(접기/펼치기)을 담당한다. 사이드바가 완전히 사라지지는 않는다.

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 한 버튼으로 사이드바 접고 펼치기 (Priority: P1)

사용자는 사이드바 안의 토글 버튼 하나를 클릭해 사이드바를 좁은 아이콘 레일로 접고, 같은 버튼을 다시 클릭해 원래 너비로 펼친다. 버튼은 접힌 상태에서도 항상 보이므로, 별도의 다른 컨트롤 없이 같은 버튼만으로 두 방향 전환이 모두 가능하다.

**Why this priority**: 이 토글 동작 자체가 기능의 본질이자 사용자가 명시적으로 요청한 핵심이다. 이것만 구현해도 "사이드바를 접었다 폈다 한다"는 가치가 완결적으로 전달되는 최소 기능(MVP)이다.

**Independent Test**: 앱에 로그인한 상태에서 사이드바의 토글 버튼을 한 번 클릭하면 사이드바가 좁은 레일로 접히고, 다시 한 번 클릭하면 원래 너비로 펼쳐지는지 확인하는 것만으로 독립 검증할 수 있다.

**Acceptance Scenarios**:

1. **Given** 사이드바가 펼쳐진 기본 상태에서, **When** 사용자가 토글 버튼을 클릭하면, **Then** 사이드바가 아이콘만 남는 좁은 레일로 접히고 본문 콘텐츠 영역이 그만큼 넓어진다.
2. **Given** 사이드바가 접힌(레일) 상태에서, **When** 사용자가 같은 토글 버튼을 다시 클릭하면, **Then** 사이드바가 원래 너비로 펼쳐지고 레이블·검색·글 목록이 다시 나타난다.
3. **Given** 어느 상태이든, **When** 사용자가 토글 버튼을 볼 때, **Then** 버튼은 항상 화면에 보이며 현재 상태(접힘/펼침)에 맞는 방향을 나타낸다.

---

### User Story 2 - 접힌 상태에서도 핵심 내비게이션 유지 (Priority: P2)

사이드바가 접힌 레일 상태에서도 사용자는 고정 내비게이션(홈, 앱 섹션의 캘린더·할 일·휴지통)과 프로필에 아이콘으로 접근할 수 있다. 각 아이콘에 마우스를 올리면 해당 항목의 이름을 툴팁으로 확인할 수 있어, 레이블이 숨겨져도 무엇인지 알 수 있다.

**Why this priority**: 접힘의 목적은 "본문에 집중할 넓은 공간"이지만, 접었다고 내비게이션이 완전히 무력화되면 다시 펼쳐야만 이동할 수 있어 가치가 반감된다. 핵심 이동 경로를 아이콘으로 유지하면 접힌 상태의 실용성이 확보된다.

**Independent Test**: 사이드바를 접은 뒤, 레일에 남은 홈 아이콘을 클릭해 홈으로 이동되고, 각 아이콘에 호버 시 이름 툴팁이 뜨는지 확인해 독립 검증할 수 있다.

**Acceptance Scenarios**:

1. **Given** 사이드바가 접힌 상태에서, **When** 사용자가 레일의 홈 아이콘을 클릭하면, **Then** 홈으로 이동한다(펼침 상태와 동일한 동작).
2. **Given** 사이드바가 접힌 상태에서, **When** 사용자가 레일의 아이콘 하나에 마우스를 올리면, **Then** 해당 항목의 이름이 툴팁으로 표시된다.
3. **Given** 사이드바가 접힌 상태에서, **When** 레일을 보면, **Then** 텍스트 기반 요소(검색 입력, 섹션 헤더, "내 글" 목록)는 숨겨지고 고정 내비 아이콘·토글·프로필 아바타만 표시된다.

---

### User Story 3 - 접힘/펼침 상태 유지 (Priority: P3)

사용자가 마지막으로 선택한 접힘/펼침 상태는 다른 글로 이동하거나 페이지를 새로고침해도 그대로 유지된다. 매번 기본값으로 되돌아가 다시 접어야 하는 번거로움이 없다.

**Why this priority**: 편의성 향상이다. 없어도 토글 자체는 동작하지만, 상태가 유지되지 않으면 이동·새로고침마다 접기를 반복해야 해 사용성이 떨어진다. 핵심 동작(P1) 이후의 개선 항목이다.

**Independent Test**: 사이드바를 접은 뒤 다른 글로 이동하거나 브라우저를 새로고침했을 때 여전히 접힌 상태로 남아 있는지 확인해 독립 검증할 수 있다.

**Acceptance Scenarios**:

1. **Given** 사이드바를 접은 상태에서, **When** 사용자가 다른 글로 이동하면, **Then** 사이드바는 접힌 상태를 유지한다.
2. **Given** 사이드바를 접은 상태에서, **When** 사용자가 페이지를 새로고침하면, **Then** 사이드바는 여전히 접힌 상태로 표시된다.
3. **Given** 사이드바를 펼친 상태에서, **When** 사용자가 새로고침하면, **Then** 사이드바는 펼친 상태로 표시된다.

---

### Edge Cases

- **접힌 상태에서 텍스트 기반 기능 접근**: 검색 입력과 "새 페이지 만들기(＋)", 글 목록은 레일에서 숨겨진다. 사용자가 이 기능을 쓰려면 먼저 사이드바를 펼쳐야 한다(레일에서는 노출하지 않는다).
- **현재 열려 있는 글의 활성 표시**: 접힌 상태에서는 "내 글" 목록이 숨겨지므로 개별 글의 활성 표시는 펼친 상태에서만 나타난다. 홈·앱 섹션 등 레일에 남는 항목의 활성 표시는 유지된다.
- **토글 버튼 연타/빠른 반복 클릭**: 빠르게 여러 번 눌러도 사이드바는 중간 상태에 갇히지 않고 마지막 클릭에 해당하는 최종 상태(접힘 또는 펼침)로 안정적으로 수렴한다.
- **좁은 화면/모바일 폭**: v1은 데스크톱 폭을 기준으로 한다. 모바일·초협폭에서의 오버레이형 사이드바 동작은 이번 범위에 포함하지 않는다(현행 동작 유지).

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 시스템은 사이드바 안에, 접힘/펼침을 전환하는 **단일 토글 버튼**을 제공해야 한다(MUST).
- **FR-002**: 사용자는 이 토글 버튼을 클릭해 사이드바를 접을 수 있어야 하며, 접힌 사이드바는 **아이콘만 보이는 좁은 레일** 형태여야 한다(MUST).
- **FR-003**: 사용자는 접힌 상태에서 **같은 토글 버튼을 다시 클릭**해 사이드바를 원래 너비로 펼칠 수 있어야 한다(MUST).
- **FR-004**: 토글 버튼은 접힘·펼침 **양쪽 상태 모두에서 항상 화면에 보여야** 하며(MUST), 별도의 다른 컨트롤 없이 이 버튼 하나로 양방향 전환이 가능해야 한다.
- **FR-005**: 토글 버튼은 현재 상태를 시각적으로 구분해 나타내야 한다(MUST) — 예: 펼침 상태에서는 "접기" 방향, 접힌 상태에서는 "펼치기" 방향.
- **FR-006**: 사이드바가 접히면 본문 콘텐츠 영역은 확보된 공간만큼 **넓어져야** 한다(MUST).
- **FR-007**: 접힌 레일에서는 텍스트 기반 요소(항목 레이블, 검색 입력, 섹션 헤더, "내 글" 글 목록)를 숨기고, **고정 내비게이션 아이콘·토글·프로필 아바타**만 표시해야 한다(MUST).
- **FR-008**: 접힌 레일의 각 내비게이션 아이콘은 펼친 상태와 **동일한 이동 동작**을 유지해야 한다(MUST) — 예: 홈 아이콘 클릭 시 홈으로 이동.
- **FR-009**: 접힌 레일에서 아이콘에 마우스를 올리면 해당 항목의 이름을 **툴팁으로 확인**할 수 있어야 한다(SHOULD).
- **FR-010**: 시스템은 사용자가 마지막으로 선택한 접힘/펼침 상태를 **유지**해, 글 간 이동 및 페이지 새로고침 후에도 동일한 상태로 표시해야 한다(SHOULD).
- **FR-011**: 접힘↔펼침 전환은 급격한 점프 없이 **부드럽게** 이루어져야 한다(SHOULD).
- **FR-012**: 토글 버튼은 **키보드로 조작 가능**해야 하며, 접근성 이름과 현재 펼침 여부를 보조 기술이 인지할 수 있어야 한다(SHOULD).

### Key Entities *(include if feature involves data)*

- **사이드바 표시 상태(Sidebar Display State)**: 사이드바가 현재 "펼침" 또는 "접힘(레일)" 중 어느 상태인지를 나타내는 사용자 UI 환경설정. 사용자별로 마지막 선택이 유지되며, 세션 간에도 보존된다. 기본값은 "펼침"이다.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 사용자는 **단일 컨트롤 한 번의 클릭**만으로 사이드바를 접거나 펼칠 수 있다(같은 버튼이 100% 양방향을 담당).
- **SC-002**: 사이드바를 접으면 본문 콘텐츠 영역의 가로 폭이 **최소 200px 이상** 넓어진다.
- **SC-003**: 접힘↔펼침 전환이 **0.3초 이내**에 완료되어 사용자가 즉각적이라고 느낀다.
- **SC-004**: 페이지를 새로고침하거나 다른 글로 이동해도 **직전에 선택한 접힘/펼침 상태가 100% 유지**된다.
- **SC-005**: 접힌 레일에서 각 내비게이션 아이콘에 마우스를 올리면 **해당 항목의 이름을 확인**할 수 있다.
- **SC-006**: 신규 사용자가 안내 없이도 토글 버튼을 발견해 사이드바를 접고 펼치는 데 성공한다(첫 시도 성공률 90% 이상).

## Assumptions

- **기존 사이드바 재사용**: 본 기능은 이미 존재하는 사이드바(`components/AppShell.tsx`의 `nav.sidebar`, 폭 264px)에 접기/펼치기 토글을 추가하는 것으로 해석한다. 사이드바를 처음부터 새로 만드는 것이 아니라, 현재 상시 노출된 사이드바에 접힘 상태를 도입한다.
- **접힌 레일 폭**: 아이콘 + 여백이 들어갈 정도의 좁은 폭(약 56px 수준)을 가정한다. 정확한 값은 계획 단계에서 `DESIGN.md`의 토큰·간격 규칙에 맞춰 확정한다.
- **상태 유지 방식**: 접힘/펼침 상태 유지는 이 앱이 이미 사용하는 클라이언트 저장 방식(로컬 지속성)을 재사용한다고 가정한다. 서버 동기화는 범위에 포함하지 않는다.
- **적용 범위**: 로그인 후 앱 셸이 렌더링되는 화면 전체에 적용된다. 로그인 화면 등 사이드바가 없는 화면은 대상이 아니다.
- **대상 폭**: v1은 데스크톱 폭 기준이다. 모바일·초협폭에서의 별도 오버레이 사이드바 동작은 범위에서 제외한다.
- **디자인 문서 동기화(의존성)**: "접힌 레일"은 `DESIGN.md`에 아직 정의되지 않은 새 패턴이다. 프로젝트 규칙(Constitution Principle III)에 따라 구현 단계에서 레일 폭·전환 모션·아이콘 정렬 등 새 디자인 결정을 `DESIGN.md`에도 함께 반영해 문서와 코드의 불일치를 방지한다.
