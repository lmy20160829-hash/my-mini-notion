# Feature Specification: Supabase 페이지 게시글 저장

**Feature Branch**: `worktree-wt1`

**Created**: 2026-07-16

**Status**: Draft

**Input**: User description: "페이지 게시글 기능 구현할거야. 이미 기존에 로컬스토리지로 구현되있었는데 내가 supabase에 만든 page 테이블에 저장되도록 변경해줘. 내가 만든 db 테이블 구조는 변경하면 안돼. 로그인된 유저만 등록 가능하고, 자신의 글만 조회 가능하고, 자신의 글만 삭제할 수 있어야돼."

## Clarifications

### Session 2026-07-16

- Q: 즐겨찾기(favorite) 기능을 어떻게 처리할까? (`page` 테이블에 favorite 컬럼이 없고 테이블 변경 금지) → A: 즐겨찾기 기능과 UI를 이번 범위에서 제거한다(테이블 변경 금지 + 단순성 원칙 준수).
- Q: 게시글 수정(제목·내용 편집) 기능을 포함할까? (사용자 명시 요구는 등록/조회/삭제였음) → A: 포함한다. 자신의 글에 한해 제목·내용을 수정할 수 있고 변경 사항이 서버에 저장된다(기존 인라인 편집 흐름 유지, US4).

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 로그인한 사용자만 게시글 작성·저장 (Priority: P1)

로그인한 사용자가 새 게시글을 작성하면, 그 글이 브라우저가 아닌 서버(공유 데이터 저장소)에 저장되어 새로고침하거나 다른 기기에서 다시 로그인해도 자신의 목록에 그대로 남아 있다. 로그인하지 않은 사용자는 글을 새로 만들 수 없다.

**Why this priority**: 로컬스토리지 → 서버 저장으로의 전환이 이 기능의 핵심 가치다. 글이 서버에 안전하게 보관되고, "로그인한 사용자만 등록"이라는 핵심 접근 제어가 여기서 성립한다. 이 스토리만 구현해도 "내 글이 서버에 남는다"는 최소 가치가 완성된다.

**Independent Test**: 로그인한 상태에서 새 글을 작성하고 페이지를 새로고침하면 그 글이 여전히 내 목록에 보인다. 로그아웃 상태에서는 글 작성 진입/시도가 막힌다.

**Acceptance Scenarios**:

1. **Given** 로그인한 사용자가 목록 화면에 있을 때, **When** 새 글을 작성하고 제목·내용을 입력하면, **Then** 그 글이 서버에 저장되고 사용자의 게시글 목록에 나타난다.
2. **Given** 로그인한 사용자가 방금 글을 작성했을 때, **When** 페이지를 새로고침하거나 다시 접속하면, **Then** 작성한 글이 그대로 유지되어 다시 표시된다.
3. **Given** 로그인하지 않은 방문자일 때, **When** 글 작성을 시도하면, **Then** 작성이 허용되지 않고 로그인이 필요함이 안내된다.

---

### User Story 2 - 자신의 글만 조회 (Priority: P1)

각 사용자는 자신이 작성한 게시글만 목록과 상세에서 볼 수 있다. 다른 사용자의 글은 어떤 화면에서도 노출되지 않으며, 로그인하지 않은 방문자에게는 게시글이 보이지 않는다.

**Why this priority**: "자신의 글만 조회"는 개인 데이터 격리라는 필수 요구사항이다. 이것이 없으면 서버 저장이 곧 데이터 유출이 되므로 P1이다. 기존 localStorage 방식의 공용/시드 데이터 노출을 대체한다.

**Independent Test**: 사용자 A가 글 몇 개를 만든 뒤, 사용자 B로 로그인하면 B의 목록에는 A의 글이 하나도 보이지 않는다. 로그아웃 상태에서는 목록이 비어 있다.

**Acceptance Scenarios**:

1. **Given** 사용자 A가 자신의 글 여러 개를 저장해 둔 상태에서, **When** 사용자 B가 로그인해 목록을 열면, **Then** B에게는 A의 글이 전혀 보이지 않고 B 자신의 글만 표시된다.
2. **Given** 로그인하지 않은 방문자일 때, **When** 목록 화면을 열면, **Then** 게시글이 표시되지 않고 로그인이 필요함이 안내된다.
3. **Given** 사용자가 자신의 글 상세를 열려고 할 때, **When** 다른 사용자의 글 식별자로 직접 접근을 시도하면, **Then** 해당 글 내용이 표시되지 않는다.

---

### User Story 3 - 자신의 글만 삭제 (Priority: P2)

사용자는 자신이 소유한 게시글만 삭제할 수 있다. 다른 사용자의 글은 삭제할 수 없다.

**Why this priority**: 삭제는 파괴적 동작이므로 소유권 강제가 중요하지만, 조회·작성이 성립한 다음에야 의미가 있으므로 P2다.

**Independent Test**: 사용자가 자신의 글을 삭제하면 목록과 서버에서 사라진다. 다른 사용자의 글에 대한 삭제 시도는 거부되어 그 글이 그대로 남는다.

**Acceptance Scenarios**:

1. **Given** 로그인한 사용자가 자신의 글을 보고 있을 때, **When** 그 글을 삭제하면, **Then** 목록과 서버에서 제거되고 새로고침 후에도 다시 나타나지 않는다.
2. **Given** 사용자가 소유하지 않은 글일 때, **When** 삭제를 시도하면, **Then** 삭제가 거부되고 그 글은 소유자에게 그대로 남는다.

---

### User Story 4 - 자신의 글 편집 저장 (Priority: P3)

사용자는 자신이 소유한 게시글의 제목과 내용을 수정할 수 있고, 변경 사항이 서버에 저장되어 유지된다.

**Why this priority**: 편집은 기존 localStorage 구현에 이미 있던 기능으로, 서버 저장 전환 시 유실되지 않도록 보존한다. 핵심 접근 제어(작성/조회/삭제) 다음의 연속성 요구라 P3다.

**Independent Test**: 자신의 글 제목·내용을 수정한 뒤 새로고침하면 변경된 내용이 유지된다.

**Acceptance Scenarios**:

1. **Given** 로그인한 사용자가 자신의 글을 열었을 때, **When** 제목이나 내용을 수정하면, **Then** 변경 사항이 서버에 저장되고 새로고침 후에도 반영되어 있다.

---

### Edge Cases

- 제목 없이(빈 제목) 글을 생성하면 어떻게 되는가? — 기존 동작(빈 제목 허용, 목록에서 "제목 없음"류로 표시)을 유지한다.
- 내용이 빈 글은 어떻게 처리되는가? — 빈 내용도 저장 가능하다(기존 동작 유지).
- 저장·조회·삭제 중 네트워크 오류가 발생하면? — 작업이 실패했음을 사용자에게 알리고, 데이터가 잘못된 상태로 남지 않도록 한다.
- 세션이 만료된 상태에서 작성/삭제를 시도하면? — 인증이 필요함을 안내하고 작업을 거부한다.
- 다른 사용자의 글 식별자로 직접 URL 접근을 시도하면? — 소유하지 않은 글은 표시하지 않는다.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 시스템은 게시글을 브라우저 로컬 저장소가 아니라 서버 측 공유 데이터 저장소에 영속화해야 한다.
- **FR-002**: 로그인한(인증된) 사용자만 새 게시글을 생성할 수 있어야 한다. 인증되지 않은 사용자의 생성 시도는 거부되어야 한다.
- **FR-003**: 생성된 각 게시글은 생성한 사용자를 소유자로 기록해야 한다.
- **FR-004**: 사용자는 자신이 소유한 게시글만 조회할 수 있어야 한다. 다른 사용자가 소유한 게시글은 목록·상세 어디에서도 노출되지 않아야 한다.
- **FR-005**: 인증되지 않은 방문자에게는 게시글이 표시되지 않아야 하며, 로그인이 필요함을 안내해야 한다.
- **FR-006**: 사용자는 자신이 소유한 게시글만 삭제할 수 있어야 한다. 소유하지 않은 게시글의 삭제 시도는 거부되어야 한다.
- **FR-007**: 사용자는 자신이 소유한 게시글의 제목과 내용을 수정할 수 있어야 하고, 변경 사항이 영속화되어야 한다.
- **FR-008**: 게시글은 최소한 제목, 내용, 생성 시각, 소유자 정보를 가져야 한다.
- **FR-009**: 시스템은 사용자가 이미 만들어 둔 기존 게시글 저장 구조(page 테이블)를 변경하지 않고 그대로 사용해야 한다. 컬럼 추가·변경·삭제는 허용되지 않는다.
- **FR-010**: 생성·조회·삭제·수정에 대한 소유권/인증 제어는 화면(UI) 차원뿐 아니라 서버(데이터 계층)에서도 강제되어야 한다. 직접 접근으로 우회할 수 없어야 한다.
- **FR-011**: 사용자가 작성·수정·삭제한 결과는 새로고침 또는 재접속 이후에도 일관되게 유지되어야 한다.
- **FR-012**: 시스템은 즐겨찾기(favorite) 기능을 제공하지 않는다. 게시글 관련 화면에서 즐겨찾기 토글·즐겨찾기 목록 등 관련 UI를 노출하지 않는다.

### Key Entities *(include if feature involves data)*

- **게시글(Page/Post)**: 사용자가 작성한 하나의 글. 주요 속성 — 제목, 내용, 생성 시각, 소유자(작성한 사용자). 소유자를 통해 조회·삭제·수정 권한이 결정된다.
- **사용자(User)**: 인증된 계정. 게시글의 소유자이며, 자신이 소유한 게시글에만 접근·조작할 수 있다.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 로그인한 사용자가 작성한 게시글은 새로고침·재접속·다른 기기 접속 후에도 100% 다시 조회된다.
- **SC-002**: 어떤 사용자에게도 다른 사용자의 게시글이 노출되지 않는다(타인 글 노출 건수 0).
- **SC-003**: 인증되지 않은 사용자의 게시글 생성 성공률은 0%다.
- **SC-004**: 사용자가 소유하지 않은 게시글의 삭제 성공률은 0%다.
- **SC-005**: 게시글을 작성한 뒤 목록/화면에 반영되기까지의 시간이 대부분의 경우(95%) 2초 이내다.
- **SC-006**: 기존 저장 구조(page 테이블)는 이 기능 구현 전후로 스키마 변경이 0건이다.

## Assumptions

- 기존 Supabase 기반 인증(구글 로그인) 기능이 이미 존재하며 이 기능은 그것을 재사용한다. 사용자를 식별하는 인증된 계정 식별자를 얻을 수 있다고 전제한다.
- 게시글 저장 구조는 사용자가 만든 `page` 테이블(식별자, 생성 시각, 제목, 내용, 소유자 식별자)이며, 이 구조를 변경하지 않고 그대로 사용한다.
- 소유자 식별자(`user_id`)는 인증된 사용자 계정과 연결된다.
- 기존 localStorage에 있던 데모/시드 게시글은 서버로 이전(마이그레이션)하지 않는다. 전환 이후 사용자는 자신의 서버 게시글(없으면 빈 목록)에서 시작한다.
- 게시글 작성 시 빈 제목·빈 내용을 허용하는 기존 동작을 유지한다.

## Out of Scope

- **즐겨찾기(favorite) 기능 전체**: 현재 localStorage 모델에는 `favorite` 속성이 있으나, 변경이 금지된 `page` 테이블에는 해당 컬럼이 없다. [Clarification 2026-07-16] 즐겨찾기 기능과 관련 UI(별표 토글·즐겨찾기 목록 등)를 이번 범위에서 제거한다. 서버 저장뿐 아니라 클라이언트 측 상태·화면 요소도 남기지 않는다.
- 게시글의 공개/공유(다른 사용자에게 보이기).
- 프로필(닉네임·아바타·소개) 관리 — 별도의 로그인/프로필 기능에서 다룬다.
- 기존 localStorage 데이터의 서버 마이그레이션.

## Dependencies

- 기존 Supabase 인증(로그인) 기능 — 인증된 사용자 식별에 필요하다.
- 사용자가 사전에 생성한 Supabase `page` 테이블 및 그 접근 제어 설정.
