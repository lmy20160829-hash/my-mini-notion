# Feature Specification: 자기소개 (Self-Introduction)

**Feature Branch**: `003-self-introduction`

**Created**: 2026-07-16

**Status**: Draft

**Input**: User description: "자기소개 기능을 구현해줘. 마이페이지에는 현재 별명 기능만 존재해. 추가로 자기소개도 등록/수정/조회 가능하도록 만들어줘. db에 introduction 컬럼 만들었으니까 연결하면 돼. 내가 만든 db변경하면 안돼."

## User Scenarios & Testing *(mandatory)*

사용자는 마이 페이지에서 프로필 정보를 관리한다. 현재는 별명(과 프로필 이미지)만 편집할 수 있다. 이 기능은 여기에 "자기소개"를 더해, 사용자가 자신을 짧게 소개하는 문장을 등록·조회·수정할 수 있게 한다.

### User Story 1 - 자기소개 등록 (Priority: P1)

자기소개가 아직 없는 사용자가 마이 페이지에서 자기소개를 입력하고 저장하면, 해당 내용이 프로필에 저장된다.

**Why this priority**: 이 스토리 없이는 기능 자체가 성립하지 않는다. 자기소개를 처음 작성해 프로필에 담는 것이 이 기능이 제공하는 핵심 가치다. 이 하나만 구현해도 "프로필에 나를 소개하는 문장을 남긴다"는 최소 가치가 완성된다.

**Independent Test**: 마이 페이지를 열어 자기소개 입력란에 문장을 입력하고 저장한 뒤, 저장 완료 표시가 나타나고 입력값이 프로필에 반영되는지 확인하면 단독으로 검증된다.

**Acceptance Scenarios**:

1. **Given** 자기소개가 비어 있는 사용자가 마이 페이지에 있을 때, **When** 자기소개 입력란에 문장을 입력하고 저장하면, **Then** 저장 완료가 표시되고 입력한 내용이 프로필에 저장된다.
2. **Given** 자기소개 입력란이 비어 있을 때, **When** 아무것도 입력하지 않고 저장하면, **Then** 자기소개는 빈 상태로 유지되고 오류 없이 저장이 완료된다.

---

### User Story 2 - 자기소개 조회 (Priority: P2)

이전에 자기소개를 저장한 사용자가 마이 페이지를 다시 열면, 저장된 자기소개가 입력란에 그대로 표시된다.

**Why this priority**: 저장한 내용을 다시 확인할 수 없다면 등록의 의미가 반감된다. 조회는 등록 다음으로 중요한, 저장 결과를 사용자에게 되돌려주는 흐름이다.

**Independent Test**: 자기소개가 저장된 상태에서 마이 페이지를 새로 로드했을 때, 저장해 둔 자기소개가 입력란에 채워져 표시되는지 확인하면 단독으로 검증된다.

**Acceptance Scenarios**:

1. **Given** 자기소개가 저장되어 있는 사용자가, **When** 마이 페이지를 다시 로드하면, **Then** 저장된 자기소개가 입력란에 표시된다.
2. **Given** 자기소개가 없는 사용자가, **When** 마이 페이지를 로드하면, **Then** 입력란은 비어 있고 안내(placeholder)만 보인다.

---

### User Story 3 - 자기소개 수정·삭제 (Priority: P3)

자기소개를 이미 저장한 사용자가 내용을 바꾸거나 비운 뒤 저장하면, 이전 값이 새 값(또는 빈 값)으로 대체된다.

**Why this priority**: 등록·조회가 동작한 뒤 자연히 필요해지는 유지·보수 흐름이다. 핵심 가치(등록)와 확인(조회) 이후의 개선 단계이므로 P3다.

**Independent Test**: 자기소개가 저장된 상태에서 입력란의 내용을 다른 문장으로 바꿔 저장하면 새 값이 반영되고, 입력란을 비워 저장하면 자기소개가 제거되는지 확인하면 단독으로 검증된다.

**Acceptance Scenarios**:

1. **Given** 자기소개가 저장되어 있을 때, **When** 다른 문장으로 바꾸어 저장하면, **Then** 이전 내용은 사라지고 새 내용이 프로필에 저장된다.
2. **Given** 자기소개가 저장되어 있을 때, **When** 입력란을 비우고 저장하면, **Then** 자기소개가 제거되어 빈 상태가 된다.

---

### Edge Cases

- 공백만 입력하고 저장하면? → 앞뒤 공백을 제거한 뒤 빈 값으로 취급한다(별명 처리 방식과 동일).
- 최대 글자 수를 초과해 입력하려 하면? → 한도까지만 입력되며 초과분은 받아들이지 않는다.
- 줄바꿈이 포함된 여러 줄 소개를 저장하면? → 줄바꿈이 보존되어 다시 조회할 때 동일하게 보인다.
- 별명만 바꾸고 자기소개는 그대로 둔 채 저장하면? → 자기소개는 변경 없이 유지된다(반대의 경우도 동일).
- 로그인하지 않은 사용자가 접근하면? → 마이 페이지 접근 자체가 기존 인증 흐름을 따르며, 이 기능은 별도의 접근 규칙을 추가하지 않는다.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 마이 페이지는 기존 별명 입력란에 더해 자기소개 입력란을 제공해야 한다(MUST).
- **FR-002**: 사용자는 자기소개를 입력하고 저장할 수 있어야 하며, 저장된 값은 프로필에 보존되어야 한다(MUST).
- **FR-003**: 마이 페이지 로드 시, 시스템은 현재 저장된 자기소개를 입력란에 표시해야 한다(MUST).
- **FR-004**: 사용자는 저장된 자기소개를 수정해 저장할 수 있어야 하며, 새 값이 이전 값을 대체해야 한다(MUST).
- **FR-005**: 사용자는 자기소개를 비워(빈 값으로) 저장해 제거할 수 있어야 한다(MUST).
- **FR-006**: 시스템은 자기소개의 앞뒤 공백을 제거해야 하며, 공백만 있는 값은 빈 값으로 취급해야 한다(MUST) — 기존 별명 처리와 동일.
- **FR-007**: 자기소개는 최대 길이(기본값 200자, Assumptions 참조)로 제한되어야 하며, 한도를 초과하는 입력은 허용하지 않아야 한다(MUST).
- **FR-008**: 자기소개는 여러 줄(줄바꿈 포함) 일반 텍스트를 지원해야 하며, 저장·조회 시 줄바꿈이 보존되어야 한다(MUST).
- **FR-009**: 저장이 성공하면 시스템은 사용자에게 저장 완료를 시각적으로 확인시켜야 하며, 이는 기존 별명 저장의 확인 방식과 일관되어야 한다(MUST).
- **FR-010**: 자기소개는 기존 프로필 저장소의 `introduction` 필드를 사용해 보존되어야 하며, 이 기능은 저장소의 스키마·구조를 변경해서는 안 된다(MUST NOT) — 사용자가 이미 만들어 둔 `introduction` 컬럼을 그대로 사용한다.
- **FR-011**: 자기소개는 선택 항목이어야 하며, 자기소개가 없는 프로필도 유효해야 한다(MUST).
- **FR-012**: 자기소개 저장은 같은 프로필의 다른 필드(별명 등)에 영향을 주지 않아야 한다(MUST).

### Key Entities *(include if feature involves data)*

- **User Profile (사용자 프로필)**: 계정의 편집 가능한 프로필 정보. 기존 속성은 별명(nickname), 프로필 이미지(avatar), 이메일(읽기 전용). 이 기능은 여기에 **자기소개(introduction)** 속성을 더한다 — 사용자를 설명하는 선택적 자유 텍스트. 이메일은 계속 읽기 전용이다.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 사용자는 마이 페이지에서 자기소개를 입력하고 저장하는 과정을 30초 이내에 완료할 수 있다(단일 입력란 + 저장 동작).
- **SC-002**: 자기소개를 저장한 뒤 마이 페이지를 다시 방문하면, 저장한 값이 100% 그대로 표시된다(재로드 후에도 유실 없음).
- **SC-003**: 자기소개를 수정해 저장하면, 이전 내용의 잔재 없이 새 값만 남는다.
- **SC-004**: 사용자는 별도 이동·조작 없이 마이 페이지 로드만으로 현재 자기소개를 확인할 수 있다.
- **SC-005**: 자기소개만 편집·저장했을 때 별명·이메일 등 다른 프로필 값은 변하지 않는다(회귀 방지).

## Assumptions

- **저장소 재사용**: 자기소개는 기존 사용자 프로필 저장소를 그대로 사용한다. 사용자가 이미 `introduction` 필드/컬럼을 만들어 두었으며, 이 기능은 그 필드를 **읽고 쓰기만** 하고 스키마를 변경하지 않는다(사용자 지시).
- **최대 길이**: 사용자가 별도로 지정하지 않아 자기소개 최대 길이를 **200자**로 가정한다(짧은 소개에 적절한 기본값). 필요 시 조정 가능하다.
- **텍스트 형식**: 서식 없는 여러 줄 일반 텍스트로 가정한다(별명과 일관, 리치 텍스트·마크다운 제외).
- **범위**: 이 기능의 범위는 **마이 페이지**에 한정한다. 자기소개를 공개 프로필·게시글 작성자 정보 등 다른 화면에 노출하는 것은 이번 범위에 포함하지 않는다.
- **저장 흐름**: 자기소개는 기존 "변경사항 저장" 동작과 저장 완료 표시를 공유한다고 가정한다(프로필 통합 저장). 별명과 함께 한 번의 저장으로 반영된다.
- **접근 모델**: 자기소개는 계정 소유자 본인만 조회·편집한다(별명과 동일한 접근 모델). 별도의 공개 조회 화면은 없다.
- **디자인 준수**: 구현 시 `DESIGN.md`의 토큰·컴포넌트·패턴(기존 `mypage-field` 패턴)을 따른다 — 계획 단계에서 반드시 `DESIGN.md`를 먼저 확인한다.

## Dependencies

- **기존 마이 페이지 및 프로필 저장소**(별명 기능)를 확장한다.
- **기존 `introduction` 저장 필드**(사용자가 생성) — 존재해야 하며, 스키마를 변경하지 않는다.

> ✅ **저장소 결정(해소됨)**: 사용자가 만들어 둔 `introduction` 컬럼은 Supabase `public.profile` 테이블에 있고, 이 워크트리는 Supabase 로그인(`004-supabase-google-login`)이 반영된 상태다. 따라서 자기소개는 **Supabase `public.profile.introduction`** 에 저장한다(별명·아바타는 기존대로 `localStorage` 오버라이드 유지 — 이번 범위에서 바꾸지 않는다). 스키마는 변경하지 않는다.
