# Quickstart: 다크 모드 검증

기능이 端to端으로 동작함을 증명하는 실행 가능한 검증 시나리오. 구현 세부는 담지 않는다.

## 사전 준비

```bash
# 워크트리 루트에서
npm install        # 최초 1회
npm test           # 유닛 테스트(자동 검증) — 전체 통과 & 무결한 출력이어야 함
npm run dev        # http://localhost:3000 (basePath 없음)
```

## 자동 검증 (Vitest)

`npm test`가 다음을 통과하면 로직 계층이 검증된다:

- `theme-resolve.test.ts` — `resolveInitialTheme` 6조합(data-model 표) 일치.
- `ThemeToggle.test.tsx` — 클릭 시 `data-theme` 전환, `aria-label` 상태 반영.
- `ThemeProvider.test.tsx` — 저장값 기반 lazy 초기화, 토글 영속화, localStorage 실패 폴백.

> FOUC(첫 페인트 전 테마)와 async 서버 레이아웃은 유닛으로 검증 불가 → 아래 수동 시나리오로 확인.

## 수동 검증 (브라우저)

로그인 후 앱 셸에서:

### S1 — 토글 전환 (US1 / SC-001, SC-005)
1. 상단 바 **중앙**의 토글 버튼을 확인한다(라이트에서 달 아이콘).
2. 클릭 → 상단바·사이드바·본문·카드가 즉시 어두운 테마로 바뀐다(300ms 내, 레이아웃 이동 없음).
3. 다시 클릭 → 라이트로 복귀. 아이콘/`aria-label`이 상태에 맞게 바뀐다.

### S2 — 선택 유지 (US2 / FR-004, FR-006)
1. 다크로 전환한다.
2. 새로고침(F5) → 다크로 다시 나타난다. **잘못된 테마 깜빡임 없음**(SC-003).
3. 탭을 닫았다 다시 연다(같은 브라우저) → 다크 유지.
4. 라이트로 명시 전환 후, OS를 "어둡게"로 바꿔도 라이트 유지(명시 선택 우선).

### S3 — 첫 방문 시스템 설정 (US3 / FR-005)
1. DevTools 콘솔에서 `localStorage.removeItem("mini-notion-theme")` 후, OS/브라우저를 "어둡게"로 설정.
2. 새로고침 → 어두운 테마로 시작한다.
3. OS를 "밝게"로 바꾸고 저장값 없는 상태에서 새로고침 → 밝게 시작.

### S4 — FOUC 없음 (FR-007 / SC-003)
1. 다크 선택 상태에서 DevTools Network를 Slow 3G로 throttle.
2. 새로고침 → 흰 화면이 번쩍인 뒤 어두워지는 현상이 **없어야** 한다(처음부터 어둡게).

### S5 — 저장 불가 폴백 (FR-010 / 엣지)
1. 브라우저를 시크릿/스토리지 차단 모드로 열거나, DevTools에서 localStorage 쓰기를 막는다.
2. 토글이 현재 세션 동안 정상 전환된다(저장만 생략, 오류로 멈추지 않음).

### S6 — 대비·가독성 (SC-004 / FR-009)
1. 다크에서 본문/제목/보조 텍스트가 배경 대비 읽기 쉬운지 확인(대비 ≥ 4.5:1은 research §R7 값으로 보장).
2. 커버 이미지·아바타·아이콘이 어두운 테마에서도 식별 가능한지 확인.

### S7 — 로그인 화면 (엣지)
1. 로그아웃하여 로그인 화면으로 이동한다.
2. 직전 선택(또는 시스템 설정)에 맞는 테마로 표시된다(토글 버튼은 없음).

## 완료 게이트 (헌법 원칙 I)

- `npm test` 전체 통과 + 에러·경고 없는 출력.
- 위 S1–S7 수동 시나리오 통과.
- `DESIGN.md`가 다크 팔레트/중앙 토글/토큰화를 반영해 코드와 일치.
