# Feature Specification: 다크 모드 (Dark Mode)

**Feature Branch**: `003-dark-mode`

**Created**: 2026-07-16

**Status**: Draft

**Input**: User description: "다크모드 기능을 구현해줘. 상단 중앙에 토글버튼을 만들고 이걸 클릭하면 다크모드로 변경돼야해"

## Clarifications

### Session 2026-07-16

- Q: 테마 선호를 어디에 저장/유지할 것인가 (로컬 기기 vs 계정 동기화)? → A: 기기/브라우저 단위 로컬 저장, 로그인 계정에 동기화하지 않음.

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 상단 중앙 토글 버튼으로 다크 모드 전환 (Priority: P1)

사용자는 화면 상단 바의 중앙에 있는 테마 토글 버튼을 눌러 인터페이스 전체를 밝은(light) 테마에서
어두운(dark) 테마로 바꾼다. 다시 누르면 밝은 테마로 돌아온다. 전환은 상단 바·사이드바·본문 등
모든 영역에 즉시 반영된다.

**Why this priority**: 이 기능의 핵심 가치 그 자체다. 상단 중앙의 토글 버튼 하나로 밝은/어두운 테마를
오가는 것만으로도 어두운 환경에서 눈의 피로를 줄이고 사용자가 선호하는 화면을 선택하게 하는 완결된
MVP가 된다.

**Independent Test**: 로그인 후 아무 화면에서나 상단 중앙 토글 버튼을 클릭해, 전체 UI가 어두운 테마로
바뀌고 다시 클릭하면 밝은 테마로 복귀하는지 확인함으로써 단독으로 검증할 수 있다.

**Acceptance Scenarios**:

1. **Given** 밝은 테마로 앱을 보고 있을 때, **When** 상단 중앙의 테마 토글 버튼을 누르면, **Then** 상단 바·사이드바·본문을 포함한 전체 인터페이스가 어두운 테마로 전환된다.
2. **Given** 어두운 테마로 앱을 보고 있을 때, **When** 같은 토글 버튼을 다시 누르면, **Then** 전체 인터페이스가 밝은 테마로 되돌아온다.
3. **Given** 어떤 앱 화면(홈, 글 편집 등)에서든, **When** 토글 버튼을 누르면, **Then** 현재 화면을 벗어나지 않고 그 자리에서 테마가 바뀐다.
4. **Given** 상단 중앙에 토글 버튼이 놓여 있을 때, **When** 사용자가 상단 바를 볼 때, **Then** 토글 버튼은 상단 바 중앙에서 명확히 식별되고 클릭할 수 있다.

---

### User Story 2 - 선택한 테마의 유지 (Priority: P2)

사용자가 한 번 선택한 테마는 페이지를 새로고침하거나 나중에 다시 방문해도 그대로 유지된다. 매번
다시 설정할 필요가 없다.

**Why this priority**: 전환 기능(P1)만으로는 방문할 때마다 다시 눌러야 해 실용성이 떨어진다. 선택의
지속성은 다크 모드가 "설정"으로서 의미를 갖게 하는 핵심 보완 요소다.

**Independent Test**: 어두운 테마를 선택한 뒤 페이지를 새로고침하고, 이어서 앱을 다시 열었을 때에도
어두운 테마가 유지되는지 확인해 단독으로 검증할 수 있다.

**Acceptance Scenarios**:

1. **Given** 어두운 테마를 선택한 상태에서, **When** 페이지를 새로고침하면, **Then** 앱은 어두운 테마로 다시 나타난다.
2. **Given** 어두운 테마를 선택하고 앱을 닫았다가, **When** 같은 기기·브라우저에서 앱을 다시 열면, **Then** 이전에 선택한 어두운 테마가 복원된다.
3. **Given** 밝은 테마를 명시적으로 선택한 상태에서, **When** 다시 방문하면, **Then** 시스템 설정과 무관하게 밝은 테마가 유지된다(명시적 선택이 시스템 기본값보다 우선한다).

---

### User Story 3 - 첫 방문 시 시스템 설정 반영 (Priority: P3)

저장된 선택이 없는 첫 방문자는, 운영체제/브라우저의 색상 테마 설정(어둡게/밝게)에 맞는 테마로 앱을
처음 보게 된다.

**Why this priority**: 사용자가 아무것도 설정하지 않아도 익숙한 환경으로 시작하게 해 첫인상을
개선한다. 핵심 전환·유지 기능이 없으면 성립하지 않으므로 후순위다.

**Independent Test**: 저장된 선택이 없는 상태에서 운영체제를 "어둡게"로 설정하고 앱을 처음 열었을 때
어두운 테마로 시작하는지, "밝게"에서는 밝은 테마로 시작하는지 확인해 단독으로 검증할 수 있다.

**Acceptance Scenarios**:

1. **Given** 저장된 테마 선택이 없고 시스템이 "어둡게"로 설정되어 있을 때, **When** 앱을 처음 열면, **Then** 어두운 테마로 시작한다.
2. **Given** 저장된 테마 선택이 없고 시스템이 "밝게"로 설정되어 있을 때, **When** 앱을 처음 열면, **Then** 밝은 테마로 시작한다.
3. **Given** 저장된 선택도 없고 시스템 설정도 확인할 수 없을 때, **When** 앱을 처음 열면, **Then** 밝은 테마(기본값)로 시작한다.

---

### Edge Cases

- **테마 깜빡임(FOUC)**: 새로고침·재방문 시 잘못된 테마가 잠깐 보였다가 바뀌는 현상 없이, 첫 렌더링부터 올바른 테마로 나타나야 한다.
- **저장소 사용 불가**: 브라우저가 선호 저장을 차단(시크릿 모드/설정 제한)한 경우에도 토글 버튼은 현재 세션 동안 정상 동작하며, 저장만 생략된다(오류로 앱이 멈추지 않는다).
- **빠른 연속 토글**: 짧은 시간에 토글 버튼을 여러 번 눌러도 최종 상태가 항상 일관되게 반영되고 시각적 잔상이나 어긋남이 없어야 한다.
- **이미지·아바타 대비**: 어두운 테마에서 커버 이미지, 아바타, 아이콘 등 시각 요소가 여전히 식별 가능해야 한다.
- **좁은 화면 배치**: 상단 바 폭이 좁아져도 중앙 토글 버튼이 좌측 워크스페이스 영역이나 우측 아이콘들과 겹치거나 잘리지 않아야 한다.
- **로그인 화면**: 상단 바(토글 버튼)가 없는 로그인 화면도 저장된 선택 또는 시스템 설정에 따른 테마를 존중한다.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 시스템은 상단 바의 중앙에 테마 토글 버튼을 제공해야 하며, 로그인 후 모든 앱 화면에서 보여야 한다.
- **FR-002**: 사용자는 이 토글 버튼을 클릭해 밝은 테마와 어두운 테마 사이를 전환할 수 있어야 한다.
- **FR-003**: 시스템은 선택된 테마를 인터페이스 전체(상단 바, 사이드바, 본문, 입력·버튼 등 컨트롤, 팝오버/다이얼로그)에 일관되게 적용해야 한다.
- **FR-004**: 시스템은 사용자의 테마 선택을 기기/브라우저 단위로 로컬 저장해, 같은 기기에서의 이후 방문과 페이지 새로고침 시 복원해야 한다(로그인 계정에 동기화하지 않음).
- **FR-005**: 저장된 선택이 없는 첫 방문에서는, 시스템은 운영체제/브라우저의 색상 테마 설정에 맞는 테마를 기본값으로 적용하고, 확인할 수 없으면 밝은 테마로 대체해야 한다.
- **FR-006**: 사용자의 명시적 선택은 시스템 설정 기본값보다 우선해야 한다.
- **FR-007**: 시스템은 인터페이스가 처음 표시되기 전에 확정된 테마를 적용해, 잘못된 테마가 깜빡이지 않도록 해야 한다.
- **FR-008**: 토글 버튼은 현재 테마 또는 누르면 일어날 동작을 명확히 나타내야 한다(식별 가능한 레이블과 아이콘).
- **FR-009**: 시스템은 두 테마 모두에서 텍스트·아이콘·이미지의 가독성과 대비를 유지해야 한다.
- **FR-010**: 저장 기능을 사용할 수 없는 환경에서도 토글 버튼은 현재 세션 동안 정상 동작해야 한다(저장 실패가 기능 실패로 이어지지 않는다).

### Key Entities *(include if feature involves data)*

- **테마 선호(Theme Preference)**: 사용자가 선택한 색상 테마(밝음/어두움)와 그것이 명시적 선택인지 시스템 기본값에서 유래했는지의 구분을 나타낸다. 기기/브라우저 단위로 저장되며, 재방문 시 적용할 테마를 결정한다.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 사용자는 어떤 앱 화면에서든 상단 중앙 토글 버튼에 대한 단 한 번의 조작(클릭/탭)으로 밝은 테마와 어두운 테마를 서로 전환할 수 있다.
- **SC-002**: 선택한 테마가 앱의 모든 화면·UI 표면의 100%에 적용되며, 반대 테마의 요소가 새어 나오지 않는다.
- **SC-003**: 재방문한 사용자는 로드 시 이전에 선택한 테마를 그대로 보며, 다른 테마가 잠깐이라도 깜빡이지 않는다(첫 렌더링부터 올바른 테마).
- **SC-004**: 두 테마 모두에서 본문 텍스트는 배경 대비 최소 4.5:1의 명암비를 만족한다.
- **SC-005**: 테마 전환은 300ms 이내에 시각적으로 완료되어 즉각적으로 느껴지며, 레이아웃 이동(layout shift)이 발생하지 않는다.

## Assumptions

- 테마 토글 버튼은 상단 바 중앙 영역(좌측 워크스페이스 제목과 우측 아이콘 컨트롤 사이의 여백)에 배치된다.
- 이 버전에서 테마는 밝음·어두움 두 가지뿐이며, 다중 팔레트나 사용자 지정 색상은 범위 밖이다.
- 테마 선호는 v1에서 기기/브라우저 단위로 저장된다(계정에 연동해 여러 기기 간 동기화하지 않는다).
- 첫 방문 기본값은 운영체제/브라우저의 색상 테마 설정을 따르고, 없으면 밝은 테마로 시작한다.
- 다크 테마의 실제 값(색·대비 등)은 프로젝트 디자인 시스템의 단일 원본(`DESIGN.md`)에 정의/추가되며, 문서와 구현이 어긋나지 않게 함께 갱신한다.
- 다크 모드는 앱 셸(상단 바가 있는 인증된 화면) 전반에 적용되며, 상단 바가 없는 로그인 화면은 토글 버튼은 없지만 저장된 선택/시스템 설정에 따른 테마를 존중한다.
