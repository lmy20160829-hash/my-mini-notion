이 프로젝트에 이미 구현되어 있는 "미니 노션"의 디자인 시스템 전체를, 하나도 빠짐없이(유실 없이) 담은 `DESIGN.md` 문서를 프로젝트 루트에 만들어줘.

이 문서 하나만 보면, 원본 코드를 보지 않고도 지금과 똑같은 화면·컴포넌트·색·간격·모션·상호작용을 그대로 재현할 수 있어야 해. "대충 정리"가 아니라 "완전 복원 가능한 명세"가 목표야.

===

## 0. 목적 (왜 만드는가)

- 이 프로젝트의 디자인은 코드 곳곳(`globals.css`의 토큰, `components/ui/*`의 프리미티브, 각 화면)에 흩어져 있어. 유지보수·재구현·핸드오프 과정에서 이 정보가 유실되기 쉬워.
- `DESIGN.md`는 그 **흩어진 디자인 메타데이터를 한 곳으로 모아 손실 없이 고정**하는 문서야.
- 그래서 이 작업의 최우선 가치는 "요약"이 아니라 "**보존(preservation)**"이야. 애매하면 줄이지 말고 남겨.

## 1. 반드시 읽어야 할 소스 (source of truth)

작성 전에 아래를 **전부 직접 열어서** 읽어. 기억이나 추측으로 쓰지 마.

- `app/globals.css` — ui:bowl 디자인 토큰(색/타이포/라운드/그림자/모션) + 모든 화면·컴포넌트 스타일. **가장 중요한 소스.**
- `components/ui/*` — Button, IconButton, Badge, Avatar, SidebarItem, SidebarSection 등 프리미티브의 props·변형(variant)·상태.
- `components/AppShell.tsx` — 상단바 + 사이드바 + 인증 가드 + 네비게이션 항목 구성.
- `app/login/page.tsx`, `app/(app)/page.tsx`, `app/(app)/posts/[id]/page.tsx`, `app/(app)/mypage/page.tsx`, `app/(app)/layout.tsx`, `app/layout.tsx` — 화면별 구조·문구·상태.
- `lib/store.tsx` — 상태 모델, localStorage 영속화, 시드 데이터, 날짜 포맷 규칙, 상호작용 로직.
- `03-reference-design.png` — 디자인 의도(레퍼런스). 실제 구현과 다르면 **구현된 코드를 정답으로 삼되**, 레퍼런스와 어긋나는 지점은 문서에 메모로 남겨.
- `02-prd.md`, `README.md` — 제품 맥락·화면 목록·의도. 디자인 결정의 "왜"를 보강할 때만 사용.

## 2. 최우선 원칙 — 유실 방지 규칙 (이걸 어기면 실패)

1. **값은 원문 그대로.** 색은 hex 그대로(`#4e97f0`), 크기는 px 그대로(`34px`), 시간은 ms 그대로(`120ms`). 반올림·근사·"약 ~" 금지.
2. **토큰은 이름과 값을 함께.** `--accent`처럼 다른 토큰을 참조하는 경우, 참조 관계와 **최종 해석된 값**을 둘 다 적어(`--accent → var(--blue-500) → #4e97f0`).
3. **전부 열거.** `globals.css`에 정의된 **모든 CSS 변수**, `components/ui`에 있는 **모든 컴포넌트**, 앱의 **모든 화면**, 사용자에게 보이는 **모든 문구**를 빠뜨리지 마. "등등", "…", "그 외 다수" 같은 말로 뭉개지 마.
4. **상태를 다 적어.** 각 인터랙티브 요소의 default / hover / active / focus / disabled / is-active / is-fav 같은 상태별 스타일을 모두 기록해.
5. **문구는 그대로 인용.** 버튼·플레이스홀더·빈 상태·안내문 등 한국어 UI 텍스트는 의역하지 말고 원문 그대로 옮겨.
6. **추적 가능하게.** 각 섹션·항목이 어느 파일(가능하면 클래스명/라인)에서 왔는지 출처를 표기해. 나중에 원본과 대조할 수 있어야 해.
7. **모르면 버리지 말고 표시.** 코드만으로 확정할 수 없는 값·의도는 임의로 지어내지 말고 `> TODO: 확인 필요 — …` 로 남겨. 침묵하며 생략하는 게 가장 나쁜 유실이야.

## 3. `DESIGN.md`가 반드시 포함해야 할 구조

아래 섹션을 모두 채워. 각 섹션에 "무엇을 담아야 하는지" 적어뒀어.

1. **개요 (Overview)**
   - 제품 한 줄 설명, 디자인 언어 이름(`ui:bowl`), 지향하는 톤(예: Notion 계열의 담백한 그래파이트 UI). 레퍼런스와 PRD에서 도출한 디자인 원칙 3~5개.

2. **파운데이션 / 디자인 토큰 (Foundations)** — `globals.css`의 `:root` 전부
   - **색상:** Neutral 램프(gray-0…900), Blue(primary, 50…700), Accent(gold/green/amber/red)를 각각 표로. 이어서 **시맨틱 매핑**(Text, Surface, Border, Accent, Status)을 `토큰명 | 참조 | 최종값 | 용도` 표로.
   - **타이포그래피:** `--font-sans`/`--font-mono` 정의(Pretendard 포함 폴백 스택 전부), 그리고 화면에서 실제로 쓰인 **타입 스케일**(예: 28/26/15/14/13/12/11px 등)을 `크기 | 굵기 | letter-spacing | line-height | 쓰인 곳`으로 정리.
   - **라운드(Radius):** xs~2xl, pill 전 값.
   - **그림자(Shadow):** xs/sm/md/lg/focus 전 값(원문 그대로).
   - **모션(Motion):** easing 2종(값 포함), duration 3종(fast/normal/slow), `@keyframes mnPop` 정의, 어떤 요소에 어떤 transition이 걸리는지.
   - **간격·레이아웃:** 주요 컨테이너 폭(사이드바 264px, 상단바 48px, list 820px, detail 760px, mypage 600px, login 400px 등)과 대표 padding/gap 값.
   - **아이콘:** 라이브러리(`lucide-react`)와 실제 사용된 아이콘 목록·기본 크기(size).

3. **컴포넌트 카탈로그 (Primitives)** — `components/ui/*` + `globals.css`의 프리미티브 클래스
   - 각 컴포넌트마다: **용도 / props / 변형(variant) / 크기 / 해부(anatomy) / 상태별 스타일(토큰 참조 포함) / 예시 마크업**.
   - 최소 포함 대상: Button(`--primary`/`--secondary`), IconButton(+`--sm`), Badge, Avatar(이미지/이니셜 폴백), SidebarItem(active/favorite 포함), SidebarSection(header/count/actions/empty), 검색 인풋(`.input-sm`), 즐겨찾기 버튼(`.fav-btn`), 업로드 버튼, 필드 인풋(`.field-input`/`.field-readonly`) 등 `globals.css`에 정의된 **재사용 조각 전부**.

4. **앱 셸 / 레이아웃 (App Shell)** — `AppShell.tsx` + `.app-root/.topbar/.sidebar/.app-main`
   - 전체 골격(상단바 + 사이드바 + 본문)의 치수·구성, 상단바 브랜드칩·워크스페이스·아이콘 버튼 순서, 사이드바(홈/검색/`내 글` 섹션/`앱` 섹션/프로필 버튼) 구성과 각 네비 항목·아이콘, 인증 가드 동작.

5. **화면별 명세 (Screens)** — 4개 화면 각각
   - 로그인(`/login`), 업무 페이지=글 목록(`/`), 글 상세(`/posts/[id]`), 마이 페이지(`/mypage`).
   - 화면마다: **레이아웃 구조 / 각 블록의 스타일과 치수 / 실제 문구 / 상태(빈 상태, 슬래시 메뉴, 저장 알림 등) / 상호작용**.
   - 놓치기 쉬운 것 반드시 포함: 목록의 빈 상태(`.empty-state`), 슬래시 커맨드 메뉴(`.slash-menu`)와 그 안의 그룹/아이템/kbd, `/page` 힌트 문구, 글 카드 구조(타일/제목/미리보기/날짜/셰브론/즐겨찾기), 상세의 브레드크럼·제목·메타·본문·삭제 버튼, 마이페이지의 아바타 행·별명/이메일 필드·저장 노트·로그아웃 카드.

6. **상호작용 · 동작 명세 (Behavior)** — `store.tsx` + 각 페이지 로직
   - `/page` 슬래시 커맨드로 새 글 생성 흐름, 제목/내용 자동 저장, 즐겨찾기 토글, 글 삭제, 로그인/로그아웃, 로그아웃 시 `/login` 리다이렉트(인증 가드), localStorage 키(`mini-notion-v1`)와 저장 스키마, 최초 방문 시 시드, 날짜 포맷 규칙("오늘/어제/N월 N일").

7. **콘텐츠 / 카피 인벤토리 (Content)**
   - 사용자에게 보이는 **모든 한국어 문자열**을 화면별로 표에 모아. 시드 데이터(기본 글 제목·본문), 기본 소유자명(`김민수`)·이메일(`minsu.kim@gmail.com`) 같은 placeholder 데이터도 포함.

8. **접근성 · 상태 디테일 (States & A11y)**
   - 포커스 링(`--shadow-focus`, `--border-focus`), placeholder 색, `::selection`, 커스텀 스크롤바(`.mn-scroll`), disabled/hover 규칙 등 상태 관련 세부.

9. **부록: 커버리지 체크리스트 (Coverage)**
   - "이 소스의 이 부분 → 문서의 이 섹션" 매핑 표. 어떤 파일의 무엇이 어디에 반영됐는지 한눈에 확인 가능하게.

## 4. 작성 형식 규칙

- 값이 많은 것(토큰·상태·문구)은 **표**로. 서술형으로 풀어 쓰다 빠뜨리지 마.
- 색 토큰 표에는 가능하면 hex를 그대로 노출(색 스와치 대용). 
- 코드 예시는 실제 클래스명·마크업 기준으로.
- 문서 언어는 한국어(값·토큰명·코드는 원문 그대로).
- 길어도 괜찮아. **완전함 > 간결함.**

## 5. 끝내기 전 자체 점검 (이 검증을 통과해야 완료)

`DESIGN.md`를 저장하기 전에 스스로 확인해:

1. `globals.css`의 `:root`에 정의된 CSS 변수를 하나씩 세어봤을 때, **전부** 문서에 등장하는가?
2. `components/ui/`의 파일 수 = 문서의 컴포넌트 항목 수와 맞는가? 빠진 프리미티브 없는가?
3. 4개 화면이 모두 있고, 각 화면의 빈 상태·특수 UI(슬래시 메뉴·저장 노트 등)까지 다뤘는가?
4. 사용자에게 보이는 한국어 문구 중 문서에 없는 게 있는가? (있으면 추가)
5. 근사·반올림한 값이 하나라도 있는가? (있으면 원문 값으로 교체)
6. 확정 못 한 항목을 `TODO`로 명시했는가? (임의로 지어내지 않았는가)

점검에서 걸린 게 있으면 고친 뒤 최종 저장해.
